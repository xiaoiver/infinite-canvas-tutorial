# Contributing

First of all, thank you for your interest in contributing to Infinite Canvas Tutorial! 🎉

This project aims to provide a step-by-step guide to building an infinite canvas from scratch, covering rendering, interaction, architecture, and related graphics concepts.

Contributions of all kinds are welcome, including:

- Fixing bugs
- Improving documentation
- Correcting typos or inaccuracies
- Adding examples
- Improving code quality
- Suggesting new tutorial topics
- Enhancing developer experience

---

## Before You Start

Please check existing Issues and Pull Requests before opening a new one.

If you plan to work on a large feature or significant change, consider opening an Issue first so we can discuss the approach before implementation.

---

## Development Setup

Clone the repository:

```bash
git clone https://github.com/xiaoiver/infinite-canvas-tutorial.git
cd infinite-canvas-tutorial
```

Install dependencies:

```bash
pnpm install
```

Start the development server:

```bash
pnpm dev
```

Build the project:

```bash
pnpm build
```

---

## Project Philosophy

This repository is primarily an educational resource.

When contributing, please prioritize:

- Clarity over cleverness
- Readability over micro-optimizations
- Educational value over abstraction
- Incremental improvements

Code should be easy for readers to understand, even if a more sophisticated implementation exists.

---

## Documentation Contributions

Documentation improvements are highly appreciated.

Examples include:

- Fixing grammar or spelling mistakes
- Clarifying explanations
- Improving diagrams
- Adding references and learning resources
- Updating outdated content

Small documentation fixes can be submitted directly as Pull Requests.

---

## Tutorial Content

When adding or modifying tutorial content:

- Keep examples focused and self-contained.
- Explain why a technique is used, not only how.
- Prefer practical examples over theoretical discussions.
- Avoid introducing unnecessary dependencies.

If a topic significantly expands the scope of the tutorial, please open an Issue for discussion first.

### Writing Guidelines

To keep the tutorial consistent:

- Use clear and concise language.
- Introduce concepts incrementally.
- Prefer diagrams and visual explanations when possible.
- Include runnable examples whenever appropriate.
- Explain trade-offs and alternative approaches.
- Link to external resources only when they provide substantial additional value.

A good tutorial section should answer:

1. What problem are we solving?
2. Why does this problem matter?
3. How does the proposed solution work?
4. What are the limitations or trade-offs?

---

## Code Style

Please follow the existing code style used throughout the repository.

General guidelines:

- Use TypeScript whenever possible.
- Keep functions small and focused.
- Prefer descriptive variable names.
- Avoid premature optimization.
- Add comments only when they improve understanding.

Format code before submitting a Pull Request.

---

## Pull Request Process

1. Fork the repository.
2. Create a feature branch:

```bash
git checkout -b feature/my-change
```

3. Commit your changes with clear commit messages.
4. Push the branch to your fork.
5. Open a Pull Request.

Please include:

- What changed
- Why the change is needed
- Screenshots or demos when applicable

Try to keep Pull Requests focused on a single topic. Smaller PRs are generally easier to review and merge.

---

## Reporting Issues

When opening an Issue, please provide:

### Bug Reports

- Environment information
- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots if applicable

### Feature Requests

- Problem statement
- Proposed solution
- Alternative approaches considered

### Tutorial Improvements

- The section or chapter involved
- What is unclear or incorrect
- Suggested improvements

---

## Code of Conduct

Please be respectful and constructive in discussions.

We welcome contributors from all backgrounds and experience levels.

Examples of positive participation:

- Giving constructive feedback
- Asking thoughtful questions
- Helping improve documentation
- Respecting differing viewpoints

Harassment, personal attacks, or disrespectful behavior will not be tolerated.

## Questions

If you're unsure whether a contribution fits the project, feel free to open an Issue and start a discussion.
